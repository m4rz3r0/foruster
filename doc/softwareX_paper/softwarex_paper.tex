\begin{abstract}
%% Text of abstract
Foruster is a cross-platform desktop application, developed in Rust, for live-system forensic analysis. Unlike traditional tools that require system shutdown, Foruster is designed to identify and catalog files of interest on active storage volumes. Its user interface, built with the Slint framework, guides the analyst through the selection of devices, the configuration of search profiles, and the real-time visualization of results. The software features heuristic detection of anomalies, such as deceptive file extensions, and ensures the integrity of findings through cryptographic hashing, optimizing the digital forensic investigation process.
\end{abstract}

\section{Motivation and Significance}

Traditional digital forensics methodology has historically centered on post-mortem analysis, which involves creating bit-by-bit disk images from powered-off systems to preserve evidence integrity. However, this ``dead-box'' approach presents significant limitations in modern computing environments. A primary drawback is the loss of volatile data—such as running processes, active network connections, or encryption keys stored in RAM—which is irretrievably lost upon shutdown and often contains crucial information for contextualizing a security incident~\cite{carrier2003open, adedayo2022review}.

The widespread use of full-disk encryption further compounds this issue. While a system is operational, encrypted volumes are mounted and their data is accessible. Once powered off, accessing this data becomes considerably more complex, if not impossible, without the decryption keys, which often only reside in volatile memory~\cite{casey2011impact}. Furthermore, the exponential growth in data volume on modern storage devices renders full imaging a time- and resource-intensive process, delaying response capabilities in time-sensitive investigations and demanding significant storage capacity~\cite{roussel2016digital, rogers2006computer}.

To overcome these obstacles, live forensics has emerged as a complementary paradigm. This technique allows investigators to collect volatile data and analyze systems while they are running, facilitating access to mounted encrypted volumes~\cite{altheide2007live}. Nonetheless, live analysis introduces its own challenges, primarily the unavoidable alteration of the system's state—a factor that must be meticulously managed and documented to maintain evidentiary validity~\cite{brezinski2005technical}. Moreover, many existing live analysis tools require a high degree of technical expertise and often rely on command-line interfaces, which can limit their adoption.

The current ecosystem of forensic tools tends towards specialization: on one hand, tools like the Volatility Framework focus exclusively on memory analysis~\cite{ligh2014art}, while platforms such as Autopsy or EnCase are designed primarily for post-mortem analysis of disk images. Although tools with live acquisition capabilities exist, such as FTK Imager, they often lack integrated functionalities for anomaly detection. Commercial solutions are typically costly and platform-specific, while open-source alternatives may present performance or compatibility trade-offs.

This landscape reveals a functional gap: the need for a tool that combines file triage on live systems with anomaly detection, packaged in a memory-safe, cross-platform, and open-source solution. Foruster was developed to address this void. Its implementation in Rust leverages the language's memory safety guarantees to prevent common vulnerabilities, such as buffer overflows, that could compromise the integrity of the forensic tool itself~\cite{jung2021rustbelt}. Its asynchronous architecture and graphical user interface aim to balance performance with accessibility.

The impact of a tool like Foruster is significant for incident response teams, security analysts, and the research community. By being open-source, it promotes transparency and reproducibility, allowing its methods to be scrutinized and validated by the community—a key tenet for the admissibility of digital evidence under rigorous legal standards like the Daubert standard~\cite{carrier2003open}.

\section{Software description}

Foruster is a desktop software that works on multiple platforms and is written in Rust. It allows live-system file investigation without needing to stop the system. Unlike usual forensic tools, Foruster works directly on active storage to find and record files that may be important, while the computer keeps running. The interface uses the Slint framework~\cite{slint_doc} and helps users choose devices, configure search settings, and see results immediately. The program also uses heuristic checks to find files with misleading extensions and protects evidence integrity with cryptographic hashes.

\subsection{Core Analysis Algorithm}

The scanning and anomaly detection workflow must balance speed and completeness. To achieve this, a set of visited paths is maintained to prevent infinite loops caused by symbolic links. The process uses a breadth-first search, which allows the scan to be stopped early if necessary while still providing partial results. Hashes (MD5, SHA-256, BLAKE3) are computed selectively for relevant or suspicious files to conserve resources. The detection logic is designed to allow legitimate polyglot files (like Office documents that are also ZIP archives), minimizing false positives. The workflow is formally described in Algorithm~\ref{alg:analysis}, which details the \texttt{analysis::walker} function.

\begin{algorithm}[htb]
\scriptsize
\caption{Live File System Triage and Anomaly Detection}
\label{alg:analysis}
\begin{algorithmic}[1]
\Require $PathsToScan$: A set of initial directory paths for analysis.
\Require $Profiles$: A set of search profiles (defined by MIME types, extensions, or file categories).
\Ensure $MatchedFiles$: A map where keys are profile names and values are sets of matching file paths.
\Ensure $SuspiciousFiles$: A set of files that matched a profile and were also flagged as anomalous.

\State $MatchedFiles \gets NewConcurrentMap()$
\State $SuspiciousFiles \gets NewConcurrentList()$

\ForAll{$path \in PathsToScan$ \textbf{in parallel}}
    \Comment{Process each starting path in a concurrent task}
    \State $batch \gets NewList()$
    \ForAll{$entry \in WalkDirectoryAsynchronously(path)$}
        \If{$entry$ is a file}
            \State $actualFormat \gets IdentifyFromContent(entry.path)$ \Comment{Reads the file header once}
            \State $suspicionReason \gets Null$
            \State $matchedProfilesForFile \gets NewList()$

            \Statex \Comment{\textit{Anomaly Detection Step}}
            \State $hiddenExt \gets CheckDeceptivePattern(entry.path)$ \Comment{e.g., "archive.jpg.exe"}
            \If{$hiddenExt$ is not Null}
                \State $suspicionReason \gets DeceptiveExtension(hiddenExt)$
            \Else
                \State $expectedKind \gets KindFromExtension(entry.extension)$
                \State $actualKind \gets actualFormat.kind()$
                \If{$IsMismatch(expectedKind, actualKind)$}
                    \State $suspicionReason \gets ContentMismatch(expectedKind, actualKind)$
                \EndIf
            \EndIf

            \Statex \Comment{\textit{Profile Matching Step}}
            \ForAll{$profile \in Profiles$}
                \If{$profile.Matches(entry.path, actualFormat)$} \Comment{Uses the already-read format}
                    \State $matchedProfilesForFile.Add(profile.name)$
                \EndIf
            \EndFor

            \State $fileResult \gets (entry.path, matchedProfilesForFile, suspicionReason)$
            \State $batch.Add(fileResult)$

            \If{$batch$ is full}
                \State ProcessBatch($batch$, $MatchedFiles$, $SuspiciousFiles$)
                \State $batch.Clear()$
            \EndIf
        \EndIf
    \EndFor
    \If{$batch$ is not empty} \Comment{Process the final batch}
        \State ProcessBatch($batch$, $MatchedFiles$, $SuspiciousFiles$)
    \EndIf
\EndFor
\State \Return $(MatchedFiles, SuspiciousFiles)$

\Statex
\Procedure{ProcessBatch}{$batch, MatchedFiles, SuspiciousFiles$}
    \ForAll{$(path, profileNames, reason) \in batch$}
        \If{$profileNames$ is not empty}
            \ForAll{$name \in profileNames$}
                \State $MatchedFiles[name].Add(path)$
            \EndFor
            \If{$reason$ is not Null} \Comment{Correlation: only add if it's also a match}
                \State $SuspiciousFiles.Add((path, reason))$
            \EndIf
        \EndIf
    \EndFor
\EndProcedure

\end{algorithmic}
\end{algorithm}


\subsection{Forensic Integrity and Chain of Custody}

Foruster follows ISO/IEC 27037:2012~\cite{iso27037} for digital evidence. In live analysis scenarios ("hot work"), the methodology must minimize the footprint on the system, preserve file metadata, and ensure traceability through a robust chain of custody. Foruster
has been designed to operate within this rigorous framework, structuring its functionalities around the forensic phases of identification, acquisition, preservation, and presentation as shown in Figure \ref{fig:forensic_phases}.

\subsubsection{Identification and Acquisition Phase}

This phase focuses on locating and collecting potential evidence. Foruster addresses this by allowing a reproducible selection of the analysis scope, where the investigator defines the storage devices and search profiles. The analysis engine (analysis::walker) operates on the file system layer, which minimizes direct interaction with the hardware and reduces the system’s footprint. This approach allows the acquisition of data from encrypted volumes (BitLocker, LUKS) that are mounted, a common scenario in active systems. Additionally, Foruster implements heuristics for identifying anomalies, such as discrepancies between the actual file type (verified using "magic numbers" with the file-format crate~\cite{carrier2005file}) and its extension, thus uncovering evidence that may have been intentionally obfuscated.

\subsubsection{Preservation and Presentation Phase}

To ensure that the evidence is not altered or contaminated, the preservation phase is critical. Foruster ensures the integrity of each identified file by generating multiple cryptographic hashes. For each relevant file, the software calculates its MD5, SHA-256, and BLAKE3 hashes~\cite{aumasson2020blake3}. This set of hashes acts as a unique digital signature of the file’s state at the exact moment of acquisition. This process is fundamental for the presentation of the evidence, as it establishes an auditable chain of custody. Any subsequent verification can recalculate these hashes to confirm that the presented evidence is an exact copy of the original.

\subsubsection{Presentation Phase}

The presentation of evidence must be clear, verifiable, and adhere to recognized standards. Foruster fulfills this phase in two ways. First, through its graphical interface, which allows the analyst a real-time visualization and filtering of the findings. Second, and more formally, by generating a final report in PDF format through the genpdfi crate. This document acts as a signed manifest of the extraction: it includes a detailed record of the operations, the analysis configuration, the characteristics of the investigated disks, and, crucially, a list of all relevant files along with their cryptographic hashes. This final report constitutes a transparent and verifiable artifact that supports the presentation of evidence in a potential judicial context.

\subsection{Implementation Considerations}

Rust's memory safety prevents crashes like buffer overflow or use-after-free. Async design with Tokio reduces CPU load on user tasks. Performance tests show less than 10\% slowdown while scanning. Cross-platform build produces native Windows and Linux binaries from one codebase. Slint gives native GUI on all platforms without heavy frameworks.

The analyst opens Foruster on the live workstation. The software displays all connected storage devices (Figure~\ref{fig:disk_selection}). In this case, the analyst selects the primary KINGSTON NVMe solid-state drive for analysis. Next, from the profiles screen (Figure~\ref{fig:profile_selection}), the analyst enables the "Videos", "Images", and "Audios" profiles to search for multimedia files that may have been exfiltrated.

When the scan begins, the interface updates in real-time. The analysis is extremely rapid, processing all 97,570 files on the target volume in just 9 seconds. The results screen (Figure~\ref{fig:results_matches}) shows that 9,336 files matched the selected profiles and 6 files were flagged as suspicious.

In the "Suspicious" tab (Figure~\ref{fig:results_suspicious}), the analyst examines the flagged files. For instance, a file named 051567.html has been identified as an anomaly. Its extension suggests it is a document, but Foruster's file header analysis reveals that its actual content is an image (image/gif). The reason provided is: "The content seems to be 'Image' but the extension indicates 'Document'." This discrepancy suggests an intentional effort to hide a file, a common anti-forensic technique to evade detection.

The analyst can then select any of these files to view detailed metadata, including creation and modification timestamps, size, and the three cryptographic hashes (MD5, SHA-256, and BLAKE3) calculated by Foruster. These hashes are crucial for verifying the file's integrity and maintaining the chain of custody.

Finally, the analyst clicks the "Export Report" button to generate a forensic report in PDF format. This document, created automatically by the software, includes the analysis time, device information, the profiles used, a summary of the findings (total files, matches, suspicious files), and a detailed list of all relevant files with their full paths and cryptographic hashes. This report serves as a formal audit trail for management or potential legal proceedings.

\subsection{Performance Evaluation}

We evaluated Foruster's performance and accuracy across multiple hardware platforms and operating systems to establish a quantitative baseline. Tests were conducted on high-end (Intel i9-14900KF), mid-range (AMD Ryzen 5 7600X), and mobile (AMD Ryzen 7 PRO 7840U) systems using the standardized Govdocs1 corpus~\cite{garfinkel2009} for benchmarking.

As detailed in Table~\ref{tab:performance}, Foruster's performance scales effectively with hardware and is significantly influenced by the operating system. On identical high-end hardware, the tool ran approximately 75\% faster on PopOS than on Windows 11, highlighting the efficiency of Linux's file I/O for this task. The tool remains highly effective even on mobile hardware, ensuring viability for in-the-field triage. Across all scenarios, resource consumption was minimal, demonstrating a low system footprint.

The anomaly detection engine's effectiveness was validated on a ground-truth 
dataset of 500 files~\cite{foruster_antiforensic_dataset} with intentionally 
mismatched extensions. The engine successfully identified all 500 files, 
achieving a perfect \textbf{100\% recall} rate for this common 
anti-forensic technique.

\subsection{Scientific and Practical Impact}

Foruster fills a gap in forensic work by letting analysts quickly examine live systems before doing full imaging. This is useful in three cases: encrypted storage, where it reads mounted volumes without complex key work; time-sensitive incidents, where immediate triage informs decisions; and limited-resource environments, where selective scanning saves storage and time.

It also helps researchers study how evidence changes over time. By looking at live analysis compared to full disk images, investigators can see what data disappears after shutdown, which gives better insight into preserving evidence.

Being open-source under GPLv3 makes the tool transparent. Many closed-source programs hide their inner workings, which can cause problems in courts that follow standards like Daubert. With open code, anyone can check the methods and reproduce results. GitHub activity shows that both researchers and practitioners are actively contributing.

Although not widely adopted yet, Foruster is already used in some labs and security operations centers for quick triage. Its open-source license makes it easier for organizations that cannot afford commercial software.

Future work includes machine-learning detection for more complex obfuscation, distributed analysis for many endpoints, CASE model integration for tool interoperability~\cite{case_dfrws_standard}, and extension to cloud and mobile systems. These updates will expand Foruster's reach while keeping live forensic capability at the core.

\section{Conclusions}

This paper presented Foruster, an open-source tool for live file triage on active systems. It addresses gaps in traditional forensics by allowing quick file discovery without shutting down the system. Key features include access to mounted encrypted volumes without complex key extraction, heuristic checks for suspicious files, a memory-safe Rust core, and automatic documentation with cryptographic hashes and PDF reports.

In tests, Foruster handled 2,269–5,819 files per second. Its extension-based anomaly detection achieved a 100\% recall rate, confirming that checking file headers against extensions works reliably. System impact was low, so it can operate discreetly during investigations.

There are some limitations. The tool requires the system to be powered on with all volumes mounted, and its heuristics may fail to detect advanced obfuscation techniques such as steganography. It does not yet support CASE standards or cloud and mobile storage.

Future work will focus on adding machine learning for smarter anomaly detection, distributed deployment for larger networks, CASE support, and cloud/mobile integration. These updates aim to keep Foruster reliable, open, and accessible to organizations with limited budgets.

By offering an open, community-verified tool, Foruster helps advance live digital forensics and supports reproducible research while lowering barriers for smaller organizations.
